# 元体知识库 — 项目实施计划

## 1. 目标与原则

- **目标**：按《01-产品与系统设计》完成「文档编辑 + 知识沉淀」核心能力，对标 ONES 的 Space/Page 与权限模型，内网部署，管理员分配权限。
- **原则**：先通过保真原型向领导汇报并争取预算，再进入开发；开发阶段先打通主流程（空间 → 页面树 → 编辑 → 版本），再扩展搜索、页面级权限、附件等；每个阶段可交付可演示。

---

## 2. 阶段划分

### 阶段 0：保真原型设计（约 2–3 周）— 预算汇报前置

**目标**：产出可演示的高保真原型与汇报材料，向领导证明本方案能对标 ONES 等市场优秀产品，支撑项目预算决策。本阶段不涉及编码与后端，仅设计 + 原型。

### 阶段 1：环境与基础（约 1 周）

| 任务 | 产出 | 负责倾向 |
|------|------|----------|
| 仓库与分支策略 | 前后端 repo 或 monorepo、main/develop | 全员 |
| 后端脚手架 | Node + Express/Fastify、PostgreSQL 连接、健康检查接口 | 后端 |
| 数据库初始化 | 执行《02-数据模型与数据库设计》建表 + 种子数据（1 个管理员、1 个空间） | 后端 |
| 前端脚手架 | React + Vite + Router + 一个空白布局页 | 前端 |
| 认证最小闭环 | 登录/登出 + Session 或 JWT + `/auth/me`，前端登录页与简单守卫 | 前后端 |

**建议实施顺序（阶段 1）**

1. 初始化后端项目与数据库连接：完成 Node + TS 脚手架、PostgreSQL 连接、`/health` 接口。  
2. 按《02》创建 `users` 表并写初始 migration，插入 1 个 system admin 账户种子数据。  
3. 选定认证方式（JWT 或 Session），实现 `/auth/login`、`/auth/logout`、`/auth/me` 以及认证中间件。  
4. 初始化前端（Vite + React + Router + React Query），实现 `/login` 页面和基础布局，并打通登录闭环（登录成功后可访问一个受保护的空白首页）。  

**里程碑**：可登录、可访问一个需认证的空白页，后端可读 DB。

### 阶段 2：空间与页面树（约 2 周）

| 任务 | 产出 | 负责倾向 |
|------|------|----------|
| 空间 CRUD API | 创建/列表/详情/更新/软删除，含权限校验 | 后端 |
| 空间权限 API | 空间权限列表、设置（用户→读/写/管理） | 后端 |
| 页面树 API | 空间下页面树（平铺/树形）、创建/更新/删除页面 | 后端 |
| 前端空间列表与进入 | 空间列表页、进入空间后占位布局（三栏骨架） | 前端 |
| 前端页面树 | 左侧页面树展示、新建/编辑/删除（含父节点选择）、拖拽排序（可选） | 前端 |
| 权限与菜单 | 根据当前用户空间权限显示「管理」等入口；无权限空间不展示 | 前端 |

**建议实施顺序（阶段 2）**

1. 后端实现 `spaces` 与基础 `pages` 表的 CRUD 接口（不含内容与版本），包括空间权限校验框架。  
2. 前端 `/` 页面使用 React Query 对接 `/spaces`，用卡片列表展示空间（参考 `web-prototype/index.html`），支持点击跳转 `/space/:spaceId`。  
3. 在 `/space/:spaceId` 页面实现三栏布局骨架，左侧用树组件展示页面列表，支持新增/重命名/删除页面（标题为主）。  
4. 接入空间权限 API，在前端根据当前用户在空间中的角色控制入口显隐（例如「Space permissions」「New page」等）。  

**里程碑**：管理员可创建空间、分配权限；用户可看到有权限的空间，进入后看到页面树并可增删改页面（标题为主，内容可占位）。

### 阶段 3：文档编辑与版本（约 2–3 周）

| 任务 | 产出 | 负责倾向 |
|------|------|----------|
| 页面内容 API | 页面详情含 content、PATCH 更新 content | 后端 |
| 版本 API | 每次保存写 page_versions、版本列表、单版本内容、回滚 | 后端 |
| 富文本编辑器集成 | 选定编辑器（如 TipTap/Quill），支持标题/列表/表格/代码块/图片/链接 | 前端 |
| 编辑与阅读切换 | 中栏阅读模式渲染 HTML、编辑模式出编辑器；顶部「编辑」按钮 | 前端 |
| 自动保存 | 防抖提交、保存状态提示、冲突或错误提示 | 前端 |
| 版本历史 UI | 版本列表、预览、回滚确认 | 前端 |
| 面包屑与 TOC | 当前页路径、右栏大纲（可选） | 前端 |

**建议实施顺序（阶段 3）**

1. 后端为 pages 增加 `content` 字段（首版使用 HTML 字符串），实现读取与更新页面内容的 API。  
2. 实施 `page_versions` 表及相关 API：在页面保存时写版本、提供版本列表与回滚接口。  
3. 前端在 `/space/:spaceId/page/:pageId` 接入富文本编辑器，实现阅读/编辑模式切换，UI 参考 `space.html` 中的 `readView/editView`。  
4. 实现自动保存逻辑（如 2 秒防抖），并用状态文案体现 `Saving... / All changes saved`。  
5. 接入版本侧栏：从 API 加载版本列表，支持查看某一版本内容与触发回滚，再刷新当前页面。  

**里程碑**：用户可在空间内打开页面、编辑富文本、自动保存、查看历史版本并回滚，完成「文档编辑 + 知识沉淀」核心闭环。

### 阶段 4：搜索与附件（约 1–2 周）

| 任务 | 产出 | 负责倾向 |
|------|------|----------|
| 搜索 API | 全局/空间内全文搜索，仅返回有权限的页面 | 后端 |
| 附件上传与存储 | 上传接口、本地或对象存储、下载/删除 | 后端 |
| 编辑器图片上传 | 粘贴或选择图片 → 调用上传 API → 插入图片 URL | 前端 |
| 搜索 UI | 顶栏或侧栏搜索框、结果列表、跳转页面 | 前端 |

**建议实施顺序（阶段 4）**

1. 后端基于 `pages` 实现全文搜索 API：先支持标题 + 正文的简单搜索，再视情况优化中文分词。  
2. 在权限校验中复用空间/页面权限逻辑，保证搜索结果只包含当前用户有读权限的页面。  
3. 实现附件上传/下载/删除接口，并确定文件存储位置（本地路径或对象存储）。  
4. 前端在顶栏实现搜索入口与 `/search` 页面，对接搜索 API，列表项点击跳转到对应空间/页面。  
5. 将编辑器中的图片上传操作接入附件 API，实现粘贴/选择图片后插入图片 URL。  

**里程碑**：支持按关键词检索、编辑时插入图片（及可选文件附件）。

---

### 阶段 5：页面级权限与打磨（约 1 周）

| 任务 | 产出 | 负责倾向 |
|------|------|----------|
| 页面权限 API | page_permissions 的增删查，校验逻辑 | 后端 |
| 页面权限 UI | 在页面设置中配置「谁可读/可写」 | 前端 |
| 权限校验贯穿 | 所有读/写接口按页面级覆盖校验 | 后端 |
| 错误与边界 | 403/404 友好页、加载态、空状态 | 前端 |
| 基础可访问性与性能 | 关键路径键盘可达、大列表/大文档基本优化 | 前端 |

**建议实施顺序（阶段 5）**

1. 按 `page_permissions` 表设计实现页面级权限 API，并在空间/页面读取逻辑中增加覆盖规则（页面级优先于空间级）。  
2. 前端基于现有 `Page settings` 弹窗接入真实数据，支持为单页配置阅读/编辑权限。  
3. 全局审查所有读/写接口，确保在服务端统一校验页面级权限，并在前端对 403 等错误给出友好提示。  
4. 补齐错误页（404/403）、加载态和空状态的统一样式，提升整体体验。  
5. 做一轮关键路径的可访问性与性能优化（键盘可达、大列表/大文档性能等）。  

**里程碑**：敏感页面可单独限制可见/可编辑；整体体验可交付内网使用。

## 3. 依赖关系简图

```
阶段0(保真原型) ──► 阶段1(环境与基础) ──► 阶段2(空间与树) ──► 阶段3(编辑与版本) ──► 阶段4(搜索与附件) ──► 阶段5(页面权限与打磨)
      汇报与预算决策              认证与 DB              页面 CRUD
```

- **阶段 0** 不依赖开发，产出原型与汇报材料；领导认可后进入阶段 1。
- 阶段 2 依赖阶段 1 的认证与 DB。
- 阶段 3 依赖阶段 2 的页面 CRUD。
- 阶段 4、5 可与阶段 3 收尾并行一部分，但建议先完成阶段 3 再全面铺开 4、5。

## 4. 风险与应对

| 风险 | 应对 |
|------|------|
| 富文本选型与后端存储格式不一致 | 在阶段 3 启动前确定「HTML 还是 JSON 块」并写进《02-数据模型》与《03-API 设计》 |
| 权限模型理解偏差 | 在阶段 2 用具体用例（如：A 仅读、B 可写、C 管理）过一遍 API 与 UI |
| 内网部署与账号来源 | 阶段 1 确定是自建用户表还是对接 LDAP/AD，避免后期大改 |
| 保真原型与开发脱节 | 阶段 0 原型严格按《01》《04》页面与布局制作，开发阶段以原型为 UI 验收基准 |
