# 测试与验收一致性验证闭环模板

把“项目实施计划 -> 开发 Todo -> 测试计划 -> 测试脚本”串成一条可复用链路，确保：
1. 每个里程碑/本周交付物都有对应的可执行测试；
2. 测试用例与 `Todo.md` 的 UI 验收清单一一对应；
3. 跑完测试后，结果会被回填到 `tests/weekN/...` 与 `docs/0N-...Todo.md`，形成闭环。

---

## 1. 四层文档与职责

### 1.1 项目实施计划（源头）
文件：`docs/05-项目实施计划.md`
职责：
- 定义阶段目标、建议顺序、阶段里程碑交付标准；
- 为后续“每周 Todo 的功能边界/对标基准（ONES）”提供依据。

### 1.2 开发 Todo（验收基准）
文件：`docs/0N-第N周开发Todo.md`
职责（本闭环强制）：
- 必须包含并维护两段 UI 保真结构：
  - `## 本周 UI 保真验收清单（开发前置，ones，硬性必达）》`
  - `## 本周 UI 验收验证（开发完成后手测，ones，硬性必达）》`
- 每个勾选项代表“必须被验证”的验收标准；
- 验收项要能落到“DOM/class/错误提示/按钮显隐/交互结果”等可测试断言上。

### 1.3 测试计划（可执行说明 + 映射表）
文件：`tests/weekN/weekN-test-plan.md`
职责：
- 把 Todo 中要验证的内容转换成：
  - 手工用例：`M-xx`
  - 自动化用例：`A-xx`
- 明确每个 `M-xx/A-xx` 的“脚本路径/执行方式/结果回填位置”。

### 1.4 测试脚本（执行器）
API 自动化：
- `tests/weekN/weekN-api-automation.mjs`
UI/E2E 自动化（可选但强烈建议用于 UI 保真项）：
- `frontend/tests/e2e/weekN.e2e.spec.mjs`

职责：
- 只实现“本周测试范围”内可自动化的断言；
- 自动化用例必须能复现（尽量用临时数据），且执行后能 best-effort 清理，避免测试污染。

---

## 2. 一致性映射规则（闭环的关键）

1. `Todo.md` 的每个“UI 验收验证”checkbox，都必须在 `weekN-test-plan.md` 中至少对应一个 `M-xx` 或 `A-xx`。
2. `weekN-test-plan.md` 中表格里的 `A-xx` 必须指向真实脚本路径（API：weekN-api-automation；UI：weekN.e2e.spec）。
3. `weekN-test-plan.md` 的 `M-xx` 必须写成“可照抄执行”的步骤（包含进入路径、检查点、预期）。
4. 测试跑完后，回填顺序固定为：
   - 先回填 `tests/weekN/weekN-test-plan.md`（结果列 + 结论）
   - 再回填 `docs/0N-第N周开发Todo.md`（对应 UI 验收验证 checkbox）

---

## 3. 推荐执行顺序（与我们 week1/week2 的经验一致）

1. 先跑该周的 API 自动化脚本（快速确认后端契约与权限策略是否正确）
2. 再跑 UI 的自动化 E2E（如果本周 UI 保真被定义为硬性必达，建议一定覆盖关键断言）
3. 最后补齐手工 M-xx（尤其是像素级/视觉细节、键盘可访问性、极端交互等）
4. 完成后按第 2 节的“回填顺序固定规则”更新两个文档

---

## 4. 测试脚本的经验规范（可直接复用）

API 自动化（`tests/weekN/weekN-api-automation.mjs`）：
- 用固定种子账号（如 `admin/admin123`、`demo/demo123`）验证权限边界；
- 覆盖本周“关键错误码/错误体字段”（配合前端 error contract）；
- 输出结果要能明确判断 Pass/Fail，并在回填时能落到测试计划的 `A-xx`。

UI/E2E 自动化（`frontend/tests/e2e/weekN.e2e.spec.mjs`）：
- 优先断言 `web-prototype` 对齐的 class/DOM（例如 `space-layout`、`sidebar`、`content-toolbar`、`tree-node`、`rightbar-section`、`toc`）；
- 断言定位要避免“同类节点 strict mode 冲突”（例如先用 `hasText` 精确命中目标节点）；
- 用临时空间数据跑验证，结束后对空间做 best-effort 清理，保证重复执行稳定。

---

## 5. 本闭环在 week1/week2 的落地体现（经验锚点）

- Week1/Week2：`tests/weekN/weekN-test-plan.md` 中把 UI 保真项拆成 `A-xx`，并指向 `frontend/tests/e2e/weekN.e2e.spec.mjs`。
- UI 验收验证 checkbox 在 `docs/0N-第N周开发Todo.md` 中按 E2E/手测结果回填，保证文档与事实一致。

---

## 6. 后续开发怎么用这份模板

每当你新增一个周的开发 Todo（`docs/0N-第N周开发Todo.md`），就按同样节奏做四件事：
1. 确认 Todo 已写对 UI 保真结构；
2. 新建/更新 `tests/weekN/weekN-test-plan.md` 并完成 `M-xx/A-xx` 映射；
3. 新建/更新对应周的 API 和（可选）E2E 脚本，保证能跑出断言结果；
4. 跑完后回填测试计划与 Todo，形成闭环。
