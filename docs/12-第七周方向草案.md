## 0. 为什么需要「不满点诊断」这一段

只写「还差很远」「体验不行」「功能不够」**无法指导排期**：  
开发不知道改哪，你也无法在评审时证明「这周值不值」。

下面给你一套 **固定维度 + 引导问题 + 空白表**。  
**满意的标准**是：任何一条不满，都能写成下面这个句子——

> **谁在什么场景下，做了什么操作，看到了什么（或没什么），因此感到什么问题；你期望变成什么样；怎样算改好了。**

## 1. 六维检查：我可能对哪里不满意？

> 不是用来「打分秀优越」，而是 **漏掉哪个维度，后面就会在哪类问题上反复踩坑**。

| 维度 | 你可能在不满什么 | 引导问题（自问） |
|------|------------------|------------------|
| **① 功能** | 缺能力、流程走不通、权限/数据不符合预期 | 有没有「必须做但系统做不了」的事？缺的是 **写文档** 还是 **管知识**（检索、归档、导出）？ |
| **② 交互与信息架构** | 找不着、步骤多、反馈弱、心智累 | 从打开产品到完成目标，**大致要几步、点哪里**？哪一步让你想关页面？ |
| **③ 语言与文案** | 中英混杂、术语乱、错误看不懂 | 第一次来的同事，**不看开发文档**能懂按钮和报错吗？ |
| **④ 视觉与一致性** | 像 demo、不专业、和原型差距大 | 哪一屏让你觉得「不像能交给团队用的产品」？ |
| **⑤ 性能与可靠** | 慢、闪、丢、刷新怪 | 是 **慢**、**错**、还是 **心里不踏实**（怕丢内容）？ |
| **⑥ 协作与运维（可选）** | 备份、审计、账号、部署 | 若明天全员用，你最怕出什么 **事故**？ |

**说明**：第 7 周默认侧重 **②③**（交互 + 中文），但若你在 **①** 或 **⑤** 上写了高优先级不满，**应以登记表为准**，本文件后面的「默认排期」必须改。

| 文章/页面管理 | 原型以空间内阅读/编辑为主；你口述「文章管理」 | 主要在 `SpacePage` 树 + 编辑 | 第 7 周采用**方案 A**：补齐「全局页面管理」独立入口页（仅 `system_admin`）+ 列表 + 跳转（不做批量/审核） |
| **文章正文富文本** | 常见 Wiki 编辑器（参考你提供的全工具栏示意） | `SpacePage` 编辑区**仅 Bold**（及既有 Insert image） | 见 **F-004** / §5.3.1 |
| **空间页顶栏与侧栏** | `space.html`：顶栏 **单行**主操作；Pages 区 **整宽**「+ New page」、树列表 | `SpacePage`：`.actions` **换行堆砌**；`sidebar-actions` **三键横挤** | 见 **F-005** / §5.3.2 |

**说明**：本表**不否定**已有后端能力（如用户、空间 API 可能存在）；不满焦点是 **产品层缺失对应页面与路径**，导致体验断裂。

## 4. 从不满到第 7 周：怎么排序才不自欺

**建议三条规则（可改，但要有规则）：**

1. **阻断型优先**：不解决就无法完成核心任务（如：保存不可靠、权限完全不可理解）→ 应先于「好看」。  
2. **高频路径优先**：每天会碰到的页面（登录、空间、编辑、搜索）→ 先于冷门设置页。  
3. **「中文优先」相关的不满**：若你强烈反感 **英文碎片、术语混乱** → 第 7 周应集中消灭 **可见文案与术语**，而不是并行开三个大功能。

**把登记表映射到周计划**：  
- 本周 P0 = **登记表里勾选「本周必须动」** 的条目（若过多，再按 §4 规则压缩；建议 ≤5 条可验收条目，**仅为防崩盘**，不是「只在乎这五条」）。  
- 其余进入 **P1 / 第 8 周**，备注里可写 **「顺延，非降级」**。

## 5. 第 7 周默认方向（在 §3 未填写前，仅作占位；填写后应覆盖）

**默认总目标（一句话）**  
在 **界面语言以简体中文为主** 的前提下，让用户在 **登录 → 空间 → 页面 → 阅读/编辑 → 搜索** 主路径上，**少困惑、少英文碎片、少无效点击**，并为后续迭代留出统一的文案与交互扩展点。

### 5.3.1 文章富文本编辑器（已登记 **F-004**）

> **目标**：对标常见富文本工具栏（你提供的示意图：**样式、标题、列表、引用、代码、链接、缩进、颜色、字体、对齐、图片/视频**等），**以可编辑、可保存、只读态可正确渲染**为底线。

**能力清单（建议分两期；第一期优先「写作必备」）**

| 分组 | 能力 | 说明 |
|------|------|------|

### 5.3.2 空间页布局与顶栏/侧栏对标 `space.html`（已登记 **F-005**）

> **参照**：`web-prototype/space.html`（约 L281–291：单列 `content-toolbar` + 单行 `.actions`；侧栏 Pages 为 **整宽** `btn btn-ghost`「+ New page」+ 树，而非多枚小按钮横排）。

**原型交互要点（摘要）**

- 顶栏：面包屑 + **一排**主操作（状态文案 + 少数白底按钮 + 一个主色 Edit）；**不把大量次级操作全部平铺**。

**P0（布局与信息层次）**

- [ ] **顶栏**：消除「无序双行」——任选或组合：`flex-wrap: nowrap` + 操作区 `min-width: 0` + **横向滚动**；**「更多 ▾」**收纳低频按钮（如 New child / Rename / Delete）；或拆 **主工具条 / 次工具条** 并统一间距。  
- [ ] **侧栏 Pages**：**对齐 `space.html` 精神**——主新建操作为 **整宽或独立一行**；Expand/Collapse 降为次要（文字链接或图标条、或移入树标题行）。  
- [ ] **与 F-001**：上述控件文案中文。

**验收**

- [ ] 在常见桌面宽度下，**顶栏操作区**不出现截图中那种杂乱换行，或第二行有**明确分组/标签**。  
- [ ] **侧栏**不再出现三枚按钮无间距挤在一行；与 `space.html` 对照可写 **一段「对齐说明」**（允许功能多于原型，但**堆砌问题须消失**）。
