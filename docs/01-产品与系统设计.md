# 元体知识库 — 产品与系统设计（对标 ONES Wiki）

## 1. 文档说明

- **项目名称**：元体知识库（YuantiWIKI）
- **对标产品**：ONES Wiki（https://ones.cn/wiki）
- **核心诉求**：文档编辑 + 知识沉淀
- **部署方式**：公司内网服务器，权限由管理员分配
- **技术栈**：前端 React、后端 Node、数据库 PostgreSQL，前后端分离

本文档描述产品定位、信息架构、功能范围与权限模型，不涉及具体实现。

---

## 2. 产品定位与目标用户

| 维度 | 说明 |
|------|------|
| **定位** | 企业内部协作文档与知识库，以「文档编辑」和「知识沉淀」为核心 |
| **主要用户** | 全员（研发、产品、运营、管理等），按管理员分配的空间/页面权限使用 |
| **使用场景** | 规范文档、技术方案、项目说明、流程手册、会议纪要等沉淀与检索 |

与 ONES 的差异：不绑定项目管理/需求/缺陷等模块，专注「文档 + 知识」，结构更轻量。

---

## 3. 信息架构（对标 ONES）

ONES 采用 **Team → Space → Page** 三级结构。本系统采用类似层级，便于权限与导航设计。

### 3.1 层级定义

```
组织（可选，单租户可省略）
  └── 空间 Space（知识库/分类）
        └── 页面 Page（文档，可多级子页面）
```

| 层级 | 说明 | 对标 ONES |
|------|------|-----------|
| **空间 Space** | 一类知识的集合，如「产品文档」「技术规范」「运维手册」。权限、目录、搜索均以空间为重要维度。 | 对应 ONES 的 Space |
| **页面 Page** | 单篇文档，支持多级子页面形成树形目录。 | 对应 ONES 的 Page |
| **组织** | 若未来多租户可增加「组织」；当前单公司内网可视为单组织，不做显式建模。 | 对应 ONES 的 Team（此处简化） |

### 3.2 空间（Space）

- **属性**：名称、描述、图标/封面（可选）、创建时间、更新时间、创建人。
- **能力**：包含一棵页面树；拥有独立权限配置（见 5. 权限模型）。
- **数量**：由管理员创建，数量无硬性限制。

### 3.3 页面（Page）

- **属性**：标题、父页面（null 表示空间根下）、顺序、内容（富文本/块存储）、创建/更新人与时间。
- **能力**：支持多级子页面（树形）；支持版本历史；支持全文检索。
- **内容形态**：富文本 + 常用块（标题、列表、表格、代码块、引用、图片等），对标 ONES 的编辑体验。

## 4. 功能范围

### 4.1 核心功能清单

| 模块 | 功能 | 优先级 | 说明 |
|------|------|--------|------|
| **空间管理** | 创建/编辑/删除空间 | P0 | 仅管理员 |
| **空间管理** | 空间列表、进入空间 | P0 | 所有有权限用户 |
| **页面树** | 页面的增删改、拖拽排序、父子关系 | P0 | 在空间内 |
| **文档编辑** | 富文本编辑、标题/列表/表格/代码块/图片 | P0 | 核心 |
| **文档编辑** | 自动保存 / 版本历史 / 回滚 | P0 | 知识沉淀必备 |
| **文档阅读** | 阅读模式、目录（TOC）、面包屑 | P0 | |
| **权限** | 空间级权限（Read Only / Edit / Admin） | P0 | 由 system_admin 分配 |
| **权限** | 页面级权限覆盖（可选） | P1 | 敏感页仅部分人可见 |
| **用户与角色** | 用户由 system_admin 维护；系统角色仅 system_admin / user；业务权限独立分配 | P0 | 内网统一账号或自建用户表 |
| **检索** | 空间内/全局搜索（标题+正文） | P1 | |
| **附件** | 图片上传、文件附件（可选） | P1 | |

### 4.2 用户与认证

- **账户来源**：Yuanti Wiki 自建用户表，由系统管理员在后台维护用户账号，而不是开放注册。
- **用户管理入口**：系统管理员通过「用户管理」后台，完成以下操作：
  - 新建用户：设置用户名、邮箱、初始密码（或发送邀请链接）；
  - 修改用户信息：如姓名、邮箱、角色；
  - 禁用/启用用户：禁用后该用户无法登录系统。
- **登录规则**：
  - 只有被系统管理员创建且处于启用状态的用户才允许登录；
  - 登录时校验用户名 + 密码（后续可扩展为与公司统一认证/SSO 对接，但首版可以先使用本地账号体系）。
- **与权限的关系**：
  - 账户管理解决「谁能登录系统」的问题；
  - 空间/页面权限配置解决「登录后能访问哪些空间和页面，以及读写级别」的问题。

### 4.3 文档编辑能力（对标 ONES）

- **块类型**：段落、标题（H1–H3）、有序/无序列表、任务列表、表格、代码块、引用、分割线、图片。
- **行内格式**：加粗、斜体、行内代码、链接。
- **扩展**：@ 提及页面（链接到其他 Page）、上传图片。
- **不纳入首版**：复杂嵌入（如看板、甘特图）、实时协同编辑（可后续迭代）。

### 4.4 明确不做的范围（与 ONES 的取舍）

- 不做了与「项目/需求/缺陷」的深度联动（无任务、看板模块）。
- 不做实时多人光标协同（可做「保存时合并」或独占编辑锁简化实现）。
- 不做对外公开分享、外链分享（内网 + 管理员分配权限即可）。

## 5. 权限模型

### 5.1 系统角色（固定两种）

| 系统角色 | 说明 | 管理端可见性 |
|----------|------|--------------|
| **system_admin** | 全局管理员 | 可见并访问顶部「管理」入口，以及用户/空间/权限管理页 |
| **user** | 普通用户 | 不可见顶部「管理」入口；不可访问空间管理或权限管理页 |

### 5.2 业务权限分配方式（由 system_admin 分配）

- **分配主体**：仅 `system_admin` 可以给普通用户分配业务权限；不采用「申请-审批」。
- **空间级权限（P0）**：每个空间下为用户指定以下三档之一：
  - `Admin`：可读 + 新建 + 编辑 + 删除（含空间内管理动作）。
  - `Edit`：可读 + 新建 + 编辑；**不可删除**。
  - `Read Only`：仅可读。
- **页面级（P1）**：某页面可单独设置可见/可编辑范围，用于覆盖空间默认权限。

### 5.3 权限校验原则

- 进入空间：需具备该空间至少「只读」。
- 查看页面：需具备空间权限，且若页面有单独权限则需在页面允许列表中。
- 新建/编辑/移动页面：需具备 `Edit` 或 `Admin`，并受页面级限制。
- 删除页面：仅 `Admin` 可执行（`Edit` 不可删除）。
- 管理端入口与路由：仅 `system_admin` 可见、可访问；普通 `user` 访问管理端返回 403。

## 6. 系统架构概览

```
[ 浏览器 ]  ←→  [ React 前端 ]  ←→  [ Node 后端 API ]  ←→  [ PostgreSQL ]
                     ↑                        ↑
                (静态资源/内网)          (认证/权限/业务)
```

- **部署**：前端构建为静态资源，与 Node 服务同域或 Nginx 反向代理；Node 连接内网 PostgreSQL。
- **认证**：内网可选用 LDAP/AD、或自建用户表 + Session/JWT，由管理员统一管理账号。

## 7. 非功能需求（设计约束）

| 项目 | 要求 |
|------|------|
| **可用性** | 内网访问，按企业级可用性要求（如 99.5%） |
| **安全** | 权限校验在服务端强制实施；敏感操作审计日志可留扩展点 |
| **性能** | 列表/树接口 < 500ms；打开单页 < 1s；搜索可异步 |
| **兼容** | 现代浏览器（Chrome/Edge/Firefox 近期版本） |
